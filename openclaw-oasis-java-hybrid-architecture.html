01｜执行摘要（给老板的 60 秒版本）

  • 现状：仿真内核高度依赖 Python OASIS 库（Agent Graph + Env + Action 执行 + DB 写入）。
  • 结论：后端可迁 Java，但仿真内核不建议一次性重写；应采用 Java+Python 混合架构。
  • 方案：Java 接管 API、任务编排、权限、可观测与报表；Python 保留 OASIS Runtime，作为仿真 Worker。
  • 收益：上线风险可控、迁移周期短、可逐步演进到更强电商仿真能力。

sequenceDiagram

loop 每一轮 Round
  OASIS->>OASIS: LLMAction 决策 + env.step()
  OASIS->>DB: 写入 trace/post/comment
  PY->>JAVA: 上报 run_state/actions 增量
end
PY-->>JAVA: 仿真完成回调
JAVA->>REP: 触发报告生成
REP-->>FE: 返回报告与可视化结果

04｜模块职责划分（边界清晰）

Java 侧（控制面）

  • 统一 API 网关、鉴权、租户与权限控制
  • 项目/任务/配置/版本管理（Config Registry）
  • 仿真任务编排、重试、幂等、审计
  • 状态聚合：run_status、timeline、agent_stats
  • 报告服务：策略复盘、指标汇总、管理看板

Python 侧（仿真面）

  • OASIS Runtime 执行（Agent Graph + Env + step）
  • 每轮动作落库与动作日志输出
  • Interview / Batch Interview 交互能力
  • 上下文注入（Profile + Memory + Event）
  • 仿真结果回调（状态、日志、错误）

06｜技术选型建议（生产可用）

后端与中间件

  • Java: Spring Boot 3 + Spring Security + Spring Data JPA
  • 任务队列: Redis Stream（轻量）或 Kafka（高吞吐）
  • 数据库: PostgreSQL（主数据）+ Redis（热状态）
  • 对象存储: MinIO / S3（报告与归档）

可观测与稳定性

  • 指标: Prometheus + Grafana（任务成功率/时延）
  • 日志: Loki/ELK（按 simulation_id 聚合）
  • 链路追踪: OpenTelemetry
  • 容错: 幂等键、重试策略、死信队列、熔断降级

08｜关键风险与治理策略

风险表现治理策略
跨语言链路复杂状态不一致、回调丢失统一 job_id/simulation_id，回调幂等，失败补偿
LLM不稳定动作质量波动、输出JSON异常结构化约束、重试、温度分层、回退策略
仿真成本升高并发高时成本不可控分层模型（核心Agent高配，长尾低配）
迁移影响业务功能回归风险灰度发布 + 双跑校验 + 指标门禁

09｜汇报结论（可直接念）

  • 我们采用 Java 控制面 + Python 仿真面，是当前成本/风险/速度最优解。
  • 该方案既保留 OASIS 成熟能力，又满足企业级后端治理要求。
  • 在电商方向，可快速扩展为“交易漏斗仿真引擎”，支持策略实验与ROI决策。
